## Bottom line

- **Yes — using a Claude Pro/Max subscription programmatically is technically possible, and is an
  officially documented Claude Code capability**, via a long-lived **subscription OAuth token**
  minted by `claude setup-token` and consumed through the `CLAUDE_CODE_OAUTH_TOKEN` env var.
  This token "authenticates with your Claude subscription and requires a Pro, Max, Team, or
  Enterprise plan. It is scoped to inference only" (source: https://code.claude.com/docs/en/authentication).
- **The subscription token is what a proxy like LiteLLM would present to Anthropic** to bill
  against the subscription instead of a pay-per-token API key. **[UNVERIFIED — LiteLLM docs not
  fetched]** that LiteLLM exposes a config path to hold/forward this token; the existence of the
  `docs.litellm.ai/.../claude_code_max_subscription` tutorial strongly implies it does, but I
  could not read it.
- **Two distinct integration directions exist and are easy to conflate** (the user correctly
  flagged this):
  1. **Claude Code → LiteLLM → Anthropic** ("LLM gateway"): Claude Code points at LiteLLM via
     `ANTHROPIC_BASE_URL`; LiteLLM holds the real credential. This is fully documented on the
     Claude Code side (source: https://code.claude.com/docs/en/llm-gateway).
  2. **LiteLLM holds the *subscription* as its upstream provider**: instead of an `sk-ant-...`
     API key, LiteLLM is configured with the subscription OAuth token as the Anthropic
     upstream. This is the `claude_code_max_subscription` tutorial's subject. **[UNVERIFIED]**.
- **Legitimacy / ToS:** The subscription→programmatic path is *officially supported by Anthropic*
  through the Agent SDK / `claude setup-token`, **but** Anthropic explicitly does **not** endorse
  or audit LiteLLM ("LiteLLM is a third-party proxy service. Anthropic doesn't endorse, maintain,
  or audit LiteLLM's security or functionality... Use at your own discretion." — source:
  https://code.claude.com/docs/en/llm-gateway). And note the upcoming quota change: **"Starting
  June 15, 2026, Agent SDK and `claude -p` usage on subscription plans will draw from a new
  monthly Agent SDK credit, separate from your interactive usage limits"** (source:
  https://code.claude.com/docs/en/authentication). So the subscription path is metered/limited,
  not unlimited.

## 1. The mechanism — how subscription auth actually works

### Subscription vs API key (the auth model)

Claude Code supports several mutually-exclusive credential types. Its **authentication precedence**
order (highest first) is (source: https://code.claude.com/docs/en/authentication):

1. Cloud-provider creds (`CLAUDE_CODE_USE_BEDROCK` / `_VERTEX` / `_FOUNDRY`).
2. **`ANTHROPIC_AUTH_TOKEN`** — sent as the `Authorization: Bearer` header. "Use this when
   routing through an LLM gateway or proxy that authenticates with bearer tokens rather than
   Anthropic API keys."
3. **`ANTHROPIC_API_KEY`** — sent as the `X-Api-Key` header. Direct Anthropic API access with a
   Console key (pay-per-token).
4. **`apiKeyHelper`** script output — for dynamic/rotating creds (e.g. vault, JWT).
5. **`CLAUDE_CODE_OAUTH_TOKEN`** — "A long-lived OAuth token generated by `claude setup-token`.
   Use this for CI pipelines and scripts where browser login isn't available."
6. **Subscription OAuth credentials from `/login`** — "the default for Claude Pro, Max, Team, and
   Enterprise users."

The key insight: **levels 5 and 6 are the *subscription* path; level 3 is the *API-key* path.**
They are different billing relationships, not just different headers.

### The subscription OAuth token (the load-bearing piece)

For non-interactive / programmatic use you mint a long-lived token (source:
https://code.claude.com/docs/en/authentication):

```bash
claude setup-token
```

> "The command walks you through OAuth authorization and prints a token to the terminal. It does
> not save the token anywhere; copy it and set it as the `CLAUDE_CODE_OAUTH_TOKEN` environment
> variable wherever you want to authenticate" (source: https://code.claude.com/docs/en/authentication).

```bash
export CLAUDE_CODE_OAUTH_TOKEN=your-token
```

Properties stated in the docs:
- It is a **one-year** OAuth token (source: https://code.claude.com/docs/en/authentication).
- "This token authenticates with your Claude subscription and **requires a Pro, Max, Team, or
  Enterprise plan**. It is **scoped to inference only** and cannot establish Remote Control
  sessions." (source: https://code.claude.com/docs/en/authentication)
- Bare mode does not read it; use `ANTHROPIC_API_KEY` or `apiKeyHelper` there (source:
  https://code.claude.com/docs/en/authentication).

**This `CLAUDE_CODE_OAUTH_TOKEN` is almost certainly the credential LiteLLM's
`claude_code_max_subscription` tutorial wires into the proxy as the Anthropic upstream**, so that
inference bills against the subscription. **[UNVERIFIED — LiteLLM docs not fetched]** — the exact
LiteLLM `config.yaml` field name (e.g. whether it goes under `litellm_params` as an
`api_key`/`auth_token`, or a dedicated `anthropic` OAuth field) was not confirmable.

### Where credentials are stored (Claude Code side)

- Linux: `~/.claude/.credentials.json`, mode `0600` (or under `CLAUDE_CONFIG_DIR` if set).
- macOS: encrypted Keychain. Windows: `%USERPROFILE%\.claude\.credentials.json`.
- "Claude Code manages `.credentials.json` through `/login` and `/logout`." Supported types
  include "Claude.ai credentials, Claude API credentials, Azure Auth, Bedrock Auth, and Vertex
  Auth." (source: https://code.claude.com/docs/en/authentication)

## 2. Concrete setup (the two directions)

### Direction A — Claude Code points at LiteLLM (the "LLM gateway")

This is the direction this project already uses for credential isolation. Claude Code is the
*client*; LiteLLM is the *gateway* that holds the real upstream credential.

Gateway requirements — LiteLLM must expose an Anthropic-compatible surface. Claude Code requires
the gateway expose at least one of: **Anthropic Messages** (`/v1/messages`,
`/v1/messages/count_tokens`), Bedrock InvokeModel, or Vertex rawPredict; and it must forward the
`anthropic-beta` and `anthropic-version` headers (source: https://code.claude.com/docs/en/llm-gateway).

Recommended endpoint — LiteLLM's unified Anthropic endpoint (source:
https://code.claude.com/docs/en/llm-gateway):

```bash
export ANTHROPIC_BASE_URL=https://litellm-server:4000
```

> Benefits over pass-through: load balancing, fallbacks, consistent cost/end-user tracking.

Auth from Claude Code to LiteLLM — three documented methods (source:
https://code.claude.com/docs/en/llm-gateway):

**Static virtual key (simplest):**
```bash
export ANTHROPIC_AUTH_TOKEN=sk-litellm-static-key
```
or in Claude Code settings:
```json
{ "env": { "ANTHROPIC_AUTH_TOKEN": "sk-litellm-static-key" } }
```
"This value will be sent as the `Authorization` header."

**Dynamic key via helper (rotating / per-user):**
```json
{ "apiKeyHelper": "~/bin/get-litellm-key.sh" }
```
```bash
export CLAUDE_CODE_API_KEY_HELPER_TTL_MS=3600000   # refresh hourly
```
"The `apiKeyHelper` has lower precedence than `ANTHROPIC_AUTH_TOKEN` or `ANTHROPIC_API_KEY`."
By default the helper is re-called after 5 minutes or on an HTTP 401 (source:
https://code.claude.com/docs/en/authentication).

**Pass-through endpoint (alternative to unified):**
```bash
export ANTHROPIC_BASE_URL=https://litellm-server:4000/anthropic
```

Optional model discovery — Claude Code can populate the `/model` picker from the gateway's
`/v1/models` (source: https://code.claude.com/docs/en/llm-gateway, env-vars):
```bash
export CLAUDE_CODE_ENABLE_GATEWAY_MODEL_DISCOVERY=1   # off by default; needs Claude Code v2.1.129+
```
Discovery authenticates the same way as inference (bearer `ANTHROPIC_AUTH_TOKEN`, else
`ANTHROPIC_API_KEY` as `x-api-key`), and only adds models whose IDs begin with `claude`/`anthropic`.
Results cache to `~/.claude/cache/gateway-models.json`.

Useful per-request headers LiteLLM can key on for attribution (source:
https://code.claude.com/docs/en/llm-gateway): `X-Claude-Code-Session-Id`,
`X-Claude-Code-Agent-Id`, `X-Claude-Code-Parent-Agent-Id`. To omit the system-prompt attribution
block (e.g. if the gateway runs its own prompt cache), set `CLAUDE_CODE_ATTRIBUTION_HEADER=0`.

> Note for non-first-party base URLs: when `ANTHROPIC_BASE_URL` is non-first-party, "MCP tool
> search is disabled by default. Set `ENABLE_TOOL_SEARCH=true` if your proxy forwards
> `tool_reference` blocks." (source: https://code.claude.com/docs/en/env-vars)

### Direction B — LiteLLM uses the *subscription* as its upstream provider

This is the subject of the `docs.litellm.ai/.../claude_code_max_subscription` tutorial. The idea:
instead of giving LiteLLM an `sk-ant-...` Console API key as the Anthropic upstream, you give it
the **subscription OAuth token** (`claude setup-token` output) so inference bills against the
Pro/Max subscription.

**VERIFIED (main agent, WebFetch of the tutorial, 2026-06-03)** — source:
https://docs.litellm.ai/docs/tutorials/claude_code_max_subscription. The documented flow is
**not** "LiteLLM holds the subscription token as upstream." Instead **Claude Code performs the
subscription OAuth itself, and LiteLLM *forwards* the client's `Authorization` (OAuth) header to
Anthropic**, while a *separate* LiteLLM virtual key (`x-litellm-api-key`) handles
tracking/controls. LiteLLM config:

```yaml
model_list:
  - model_name: anthropic-claude
    litellm_params:
      model: anthropic/claude-sonnet-4-20250514
general_settings:
  forward_client_headers_to_llm_api: true   # MANDATORY — lets the client OAuth token reach Anthropic
```

Claude Code side:
```bash
export ANTHROPIC_BASE_URL=http://localhost:4000
export ANTHROPIC_MODEL="anthropic-claude"
export ANTHROPIC_CUSTOM_HEADERS="x-litellm-api-key: Bearer sk-<litellm-virtual-key>"
# then `claude` → select "Claude account with subscription" → authorize in browser
```
Two auth headers: `x-litellm-api-key` (authenticates to LiteLLM, for tracking/limits) and
`Authorization: Bearer <oauth>` (the subscription token, forwarded to Anthropic).

**Important correction to the token-location question:** in *this* documented variant the
**subscription OAuth token lives in the Claude Code client (i.e. in the sandbox)** and is merely
forwarded by LiteLLM — it does **not** sit only in the proxy. So there are two sub-variants:

- **B1 — forward client token (verified, documented):** token in the sandbox; LiteLLM forwards +
  meters via its virtual key. Simpler, but the high-value subscription token is in the sandbox.
- **B2 — proxy holds the token (the cleaner-isolation idea; NOT shown in this tutorial):** mint
  `CLAUDE_CODE_OAUTH_TOKEN` via `claude setup-token` and configure it as LiteLLM's Anthropic
  upstream so the token never enters the sandbox. Whether LiteLLM supports supplying the OAuth
  token as a static upstream credential (vs. only forwarding the client header) is **still
  [UNVERIFIED]** — the `byok` tutorial (not yet fetched) may cover it.

For this project's credential-isolation goal, **B2 is preferable** but unconfirmed; **B1 is
confirmed** but puts the token in the sandbox (protected only by the sandbox boundaries). This is
a real decision — see §6.

## 3. Subscription vs API key — practical difference

| | API key (`ANTHROPIC_API_KEY`) | Subscription (`CLAUDE_CODE_OAUTH_TOKEN` / `/login`) |
|---|---|---|
| Billing | Pay-per-token, Console org | Flat subscription (Pro/Max/Team/Enterprise) |
| Header | `X-Api-Key` | `Authorization: Bearer` |
| How obtained | Claude Console | `claude setup-token` (OAuth) / interactive `/login` |
| Programmatic use | Always works | Requires the long-lived OAuth token; **scoped to inference only** |
| Limits | Org rate/spend limits | Subscription usage limits; **from 2026-06-15 a separate monthly "Agent SDK credit" for `claude -p`/Agent SDK** |

Sources: https://code.claude.com/docs/en/authentication , https://code.claude.com/docs/en/env-vars

- Does the subscription path avoid per-token billing? **Yes in principle** — it draws on the
  subscription rather than a metered Console key. But it is **not unlimited**: subscription usage
  limits apply, and "**Starting June 15, 2026, Agent SDK and `claude -p` usage on subscription
  plans will draw from a new monthly Agent SDK credit, separate from your interactive usage
  limits**" (source: https://code.claude.com/docs/en/authentication). Exact credit size /
  overage behavior is in the linked support article, which I could not fetch
  (https://support.claude.com/en/articles/15036540-use-the-claude-agent-sdk-with-your-claude-plan).
- Precedence gotcha: "If you have an active Claude subscription but also have `ANTHROPIC_API_KEY`
  set... the API key takes precedence once approved." Run `unset ANTHROPIC_API_KEY` and check
  `/status` to confirm which method is active (source: https://code.claude.com/docs/en/authentication).

## 4. Direction of integration — the answer to the user's core ambiguity

Both directions are real and they **stack**:

```
[agent / Claude Code]  --virtual key (ANTHROPIC_AUTH_TOKEN)-->  [LiteLLM :4000]  --subscription OAuth token-->  [Anthropic]
   Direction A (llm-gateway docs)                                   Direction B (claude_code_max_subscription docs)
```

- The `code.claude.com/llm-gateway` doc is **Direction A**: how Claude Code is configured to
  treat LiteLLM as its gateway (`ANTHROPIC_BASE_URL`, `ANTHROPIC_AUTH_TOKEN`/`apiKeyHelper`).
- The LiteLLM `claude_code_max_subscription` doc is **Direction B**: how LiteLLM is configured to
  use the subscription as its Anthropic upstream. **[UNVERIFIED]**.
- The LiteLLM `claude_code_byok` doc is the LiteLLM-side mirror of Direction A (bring an API key
  *or* token, mint virtual keys for clients). **[UNVERIFIED]**.

For this project you want **both at once**: agents use virtual keys to reach LiteLLM (A), and
LiteLLM's Anthropic upstream is the subscription token (B).

## 5. Caveats, legitimacy, ToS, risk

- **LiteLLM is third-party and unendorsed by Anthropic.** Verbatim: "LiteLLM is a third-party
  proxy service. Anthropic doesn't endorse, maintain, or audit LiteLLM's security or
  functionality. This guide is provided for informational purposes and may become outdated. Use
  at your own discretion." (source: https://code.claude.com/docs/en/llm-gateway)
- **Supply-chain warning (concrete, recent).** "LiteLLM PyPI versions 1.82.7 and 1.82.8 were
  compromised with credential-stealing malware. Do not install these versions." Remediation:
  remove the package, rotate all credentials, follow BerriAI/litellm#24518 (source:
  https://code.claude.com/docs/en/llm-gateway , https://github.com/BerriAI/litellm/issues/24518).
  This is directly relevant to a project whose whole point is credential isolation: a compromised
  proxy holding a subscription OAuth token is a high-value target. Pin/verify LiteLLM versions.
- **Is the subscription→programmatic path itself legitimate?** Yes — `claude setup-token` and the
  `CLAUDE_CODE_OAUTH_TOKEN` env var are first-party, documented features explicitly intended for
  "CI pipelines and scripts" (source: https://code.claude.com/docs/en/authentication). Using that
  token from inside LiteLLM is the gray area: it's not a documented Anthropic integration, only a
  LiteLLM community tutorial. **I could not read that tutorial**, so I can't confirm whether
  LiteLLM does anything beyond forwarding the bearer token, nor whether Anthropic's subscription
  ToS permits proxying it across multiple agents/users. **Flag as uncertain; verify ToS.**
- **Quota/metering.** Subscription programmatic use is metered and, from 2026-06-15, carved into a
  separate monthly Agent SDK credit (source: https://code.claude.com/docs/en/authentication).
  Fan-out across many sandboxed agents could exhaust this quickly — budget accordingly.
- **Token lifetime / rotation.** The OAuth token is ~1 year; plan rotation. `apiKeyHelper` +
  `CLAUDE_CODE_API_KEY_HELPER_TTL_MS` is the documented mechanism for rotating creds on the
  *client→gateway* hop, but the *gateway→Anthropic* subscription token rotation is a LiteLLM
  concern (unverified).
- **Stability.** Anthropic notes the LiteLLM guide "may become outdated"; treat the integration as
  best-effort, not contractual.

## 6. How this maps to `llm-system-containers`

Current design (source: `planning/services/llm-proxy.md`, `planning/security-model.md`):
LiteLLM runs as `svc-litellm` on `:4000` in its own L2 container; agents in sandbox containers get
**virtual keys**; "real provider credentials live only in the proxy service"; network policy
should let agents reach **only** the proxy for LLM egress.

Wiring for the subscription path:

- **Token placement = the proxy container only.** The subscription OAuth token
  (`CLAUDE_CODE_OAUTH_TOKEN` from `claude setup-token`) is the "real credential." Per the
  project's credential-isolation model it belongs **only inside `svc-litellm`** — not in the
  sandbox, not handed to agents. Agents continue to see *only* virtual keys. This preserves the
  backstop: "even an agent that fully escapes its own permission framework still only ever sees a
  virtual key" (source: `planning/services/llm-proxy.md`).
- **Agent/sandbox side (Direction A):** each agent's Claude Code (or Agent SDK) sets
  `ANTHROPIC_BASE_URL=http://svc-litellm:4000` and `ANTHROPIC_AUTH_TOKEN=<virtual-key>`. No
  Anthropic credential ever enters the sandbox.
- **Proxy side (Direction B):** `svc-litellm` is configured with the subscription token as its
  Anthropic upstream (exact `config.yaml` field **[UNVERIFIED]** — confirm against the LiteLLM
  tutorial). LiteLLM's per-key budgets/rate-limits then bound each agent's draw on the shared
  subscription quota — important given the 2026-06-15 Agent SDK monthly credit.
- **Where the token lives — explicit answer to the user's question:** in the **proxy** container,
  never the sandbox and (ideally) never the host long-term. Mint it once via `claude setup-token`,
  inject it into `svc-litellm` as a secret/env, and keep it out of images and the host
  filesystem.
- **Observability still works:** LiteLLM logging hooks → Phoenix are unchanged; the
  `X-Claude-Code-*` session/agent headers (source: https://code.claude.com/docs/en/llm-gateway)
  can be logged for per-agent attribution without parsing request bodies — a nice fit for the
  per-UID agent-profile model.
- **Risk note for this project specifically:** because the subscription token is long-lived and
  high-value, the LiteLLM supply-chain warning above is material. Pin LiteLLM to a vetted version,
  keep `svc-litellm` in its own L2 container (already the plan), and apply Tetragon egress rules so
  even the proxy can only reach Anthropic.

Open project decisions this intersects with (`planning/services/llm-proxy.md` open items): key
issuance lifecycle, budget/rate-limit defaults (now doubly relevant due to subscription quota),
and the "agents reach only the proxy" network policy.
